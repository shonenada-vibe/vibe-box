# Vibe Box 项目计划

## 项目目标

基于 `ESP32-S3-Touch-ePaper-1.54` V2 开发一台独立设备，参考 `/Users/shonenada/Projects/yesnoaudio` 的交互思路，但把整体架构从：

- `板子发 BLE 控制信号`
- `电脑录音 / 调用 STT / 处理文本`
- `电脑侧输出结果`

改成：

- `板子本地采音`
- `板子通过 Wi-Fi 直接请求服务器`
- `板子本地在墨水屏显示结果`
- `可选：板子本地通过 ES8311 + 扬声器播放语音`

核心要求是摆脱电脑，设备单独联网即可工作。

## 已确认的硬件与开发前提

- 你的板子是 `ESP32-S3-PICO-1-N8R8` 版本，也就是 Waveshare 文档中的 `V2` 版本。
- 官方文档说明 `V2` 与 `V1` 示例程序不通用，必须使用对应版本资源。
- 官方文档说明这块板子支持 `ESP-IDF` 和 `Arduino`，其中 `ESP-IDF` 版本要求 `v5.5.0+`。
- 官方示例已经覆盖：
  - 墨水屏
  - Wi-Fi STA
  - 音频测试
  - SHTC3 温湿度
  - RTC / 低功耗
  - 触摸测试

结论：这个项目应优先基于 `ESP-IDF`，不要从 `yesnoaudio` 的 Zephyr BLE 固件直接移植。

## 参考项目可复用与不可复用部分

`yesnoaudio` 当前结构：

- 设备端 `src/main.c` 主要负责 BLE 按键/录音控制通知
- 主逻辑在 `host/app_core.py`
- 真正的录音、转写、翻译都在电脑端完成

可复用的只有“产品流程”和“服务端能力划分”：

- 录音开始 / 录音结束状态机
- STT / 翻译 / 后处理链路
- 配置项组织方式
- 错误处理与运行态状态字段

不应复用的部分：

- BLE 作为主通信通道
- 电脑麦克风录音
- macOS 键盘注入
- curses / TUI / 菜单栏相关逻辑

## 推荐目标架构

本项目采用 `B` 方案：

- 设备端保留主要产品逻辑
- 自建一个很薄的 `server`
- `server` 只负责鉴权、转发到第三方 AI 服务、压缩结果返回给设备

明确边界：

- 设备端负责：
  - 采音
  - 交互
  - 状态机
  - 墨水屏显示
  - 配网
  - 本地缓存
  - 错误恢复
- 薄服务端负责：
  - 保存 API key
  - 接收设备音频
  - 调用第三方 STT / LLM / TTS
  - 返回适合 200x200 屏幕的短结果

不做的事：

- 不把主交互流程放在 server
- 不做桌面端中转
- 第一版不做复杂后台管理系统
- 第一版不做重状态、多租户、数据库驱动的复杂后端

### 设备端

- `ui_epaper`
  - 启动画面
  - Wi-Fi 配网状态
  - 待机页
  - 录音中页
  - 上传中页
  - 结果展示页
  - 错误页
- `audio_input`
  - ES8311 初始化
  - 麦克风采样
  - PCM 缓冲
  - 可选 WAV 封装
- `network_client`
  - Wi-Fi STA 连接
  - HTTPS 请求
  - 可选 WebSocket
  - 失败重试
- `server_protocol`
  - 请求打包
  - 响应解析
  - 版本兼容
- `device_state`
  - 当前页面
  - 当前网络状态
  - 电量 / 温湿度 / RTC
  - 最后一条识别结果
- `storage`
  - NVS 保存 Wi-Fi、服务器地址、token、设备配置
- `input`
  - 触摸 / 按键触发录音
  - 长按进入配网或设置
- `power`
  - 局部刷新
  - 休眠唤醒
  - 周期性传感器刷新

### 服务端

- `api_gateway`
  - 设备鉴权
  - 接收音频
  - 转发到第三方 STT / LLM / TTS
- `provider_adapter`
  - 对接外部语音识别和文本生成 API
- `response_shaper`
  - 把长文本压缩成适合 200x200 墨水屏显示的短文本
- `tts_proxy`（可选）
  - 返回音频 URL 或二进制音频
- `device_config_api`（可选）
  - 远程下发展示模板、提示词、功能开关

服务端设计原则：

- 无状态优先
- 只保留极少数接口
- 不存原始音频，除非调试明确需要
- 把云服务密钥留在服务端，不下发到设备

## 通信方案建议

第一版不要直接做全双工语音助手，先做简单稳定的 `HTTPS REST`。

这里的关键点不是“有没有 server”，而是“server 要足够薄”：

- 设备仍然是主系统
- server 只是安全和云能力的适配层

推荐第一版流程：

1. 设备按键或触摸触发录音
2. 录制 `3-8` 秒单声道音频
3. 设备通过 `HTTPS POST /v1/query` 上传音频和设备元数据
4. 薄服务端调用第三方能力并返回 JSON：
   - `transcript`
   - `reply_text`
   - `display_lines`
   - `speak_text` 或 `audio_url`
5. 设备刷新墨水屏显示结果
6. 如开启语音播报，再拉取音频并播放

建议先不用 WebSocket 的原因：

- 固件复杂度更低
- 更容易定位问题
- 更适合按一下说一句的交互
- 对墨水屏刷新模式更友好

后续如果要做连续对话，再升级到：

- WebSocket
- 分段上传
- 流式转写
- 流式 TTS

## 设备与服务器接口草案

### 请求

`POST /v1/query`

表单字段或 JSON + 二进制音频：

- `device_id`
- `firmware_version`
- `session_id`
- `temperature`
- `humidity`
- `battery_level`
- `language`
- `audio_format`
- `audio`

### 返回

```json
{
  "request_id": "req_123",
  "transcript": "今天上海天气怎么样",
  "reply_text": "今天多云，气温 24 到 29 度。",
  "display_lines": [
    "今天多云",
    "24-29C",
    "出门可带伞"
  ],
  "speak_text": "今天上海多云，气温二十四到二十九度。",
  "refresh_mode": "partial",
  "cache_ttl_s": 60
}
```

设备端约束：

- 屏幕只直接依赖 `display_lines`
- `reply_text` 可用于翻页详情
- `speak_text` 可为空
- 薄服务端负责压缩输出长度，避免设备端做复杂排版

## 推荐的产品 MVP

第一阶段先把目标收紧，不要一开始把所有硬件都打开。

### MVP 必做

- 墨水屏显示系统状态与文本结果
- Wi-Fi 配网并保存
- 触摸或按键触发一次录音
- 上传音频到薄服务端
- 返回文本并显示
- 显示温湿度

### MVP 暂缓

- 本地唤醒词
- 连续对话
- 本地大模型
- 蓝牙配网 App
- TF 卡多媒体
- 复杂多页面 UI
- 本地离线 STT

### MVP 完成标准

- 不接电脑也能启动
- 上电后自动联网
- 用户触发后 10 秒内看到结果
- 弱网或失败时屏幕能明确显示错误原因

交付物：

## Phase 1: 工程骨架

目标：建立独立固件工程，不再依赖电脑侧运行时，并明确“设备优先”的职责边界。

任务：

- 初始化 ESP-IDF 项目
- 规划组件目录
- 建立统一日志、错误码、配置读取
- 建立设备状态机：
  - boot
  - provisioning
  - idle
  - recording
  - uploading
  - displaying
  - error
- 接入 NVS 保存配置

建议目录：

```text

  main/
  components/

    audio_input/
    net_client/
    server_protocol/
    sensors/
    storage/
    power/
```

交付物：

- 空跑主循环
- 墨水屏状态页切换

## Phase 2: 配网与设备配置

目标：让设备脱离电脑后也能入网。

任务：

- 实现首次配网流程
- 推荐优先级：
  1. SoftAP + 简单网页配网
  2. 备用串口配置
  3. 后续再考虑 BLE 配网
- 保存以下配置到 NVS：
  - Wi-Fi SSID / 密码
  - 服务器地址
  - API token
  - 语言
  - 录音时长
- 墨水屏展示配网状态和 IP

交付物：

- 不接电脑完成配网
- 重启后自动重连 Wi-Fi

## Phase 3: 音频链路

目标：打通设备本地采音。

任务：

- 初始化 ES8311
- 配置 I2S 录音
- 选择统一输入格式：
  - `16kHz`
  - `mono`
  - `16-bit PCM`
- 录音缓存优先放在 PSRAM
- 支持按住录音或点击开始/结束二选一
- 生成上传用的裸 PCM 或 WAV
- 如需要，加入简单 VAD 或静音截断

交付物：

- 板子本地可录到一段有效音频
- 串口日志可确认时长与大小

## Phase 4: 薄服务端最小实现

目标：尽快完成“设备 -> 薄服务端 -> 第三方 AI 能力”的闭环。

任务：

- 新建最小薄服务端
- 先只做一个接口：`POST /v1/query`
- 服务端内部先用最简单策略：
  - 收音频
  - 转发到一个第三方 STT
  - 可选再转发到一个 LLM
  - 返回屏幕友好的短文本
- 可优先参考 `yesnoaudio` 里的：
  - 转写 provider 选择思路
  - 文本后处理思路
  - 配置结构思路
- 服务端不负责：
  - 本地录音
  - BLE
  - UI
  - 会话主状态机
  - 设备页面编排

交付物：

- 本地或云端薄服务端可返回 JSON 结果

## Phase 5: 设备联网请求

目标：让设备真正独立请求薄服务端。

任务：

- 集成 `esp_http_client`
- 加入 TLS 证书校验
- 处理超时、重试、断网恢复
- 音频上传
- JSON 解析
- 将响应映射到设备 UI 状态

交付物：

- 按一次按钮，服务器收到请求
- 返回内容显示到墨水屏

## Phase 6: 墨水屏体验优化

目标：让显示结果可读、可用、刷新成本合理。

任务：

- 设计 200x200 的文本布局
- 字体选型：
  - 中文主字体
  - 英文 / 数字辅助字体
- 支持：
  - 标题
  - 主内容
  - 底部状态栏
- 区分全刷与局刷
- 避免频繁刷新导致闪屏和耗电

推荐页面：

- 首页：时间 / 温湿度 / 网络状态
- 录音页：进度或提示语
- 结果页：3-6 行核心文本
- 错误页：错误码 + 简短原因

交付物：

- 基本页面稳定显示
- 单次问答刷新次数可控

## Phase 7: 传感器与扩展信息

目标：把板载传感器整合进产品，而不只是 demo。

任务：

- 定时读取 SHTC3
- 在首页显示温湿度
- 可选把温湿度作为请求上下文传给服务器
- 接入 RTC，提供时间显示
- 后续可加入：
  - 电池电量
  - 上次同步时间

交付物：

- 首页能长期稳定显示环境信息

## Phase 8: 语音回放

目标：利用 ES8311 和扬声器提升交互完成度。

任务：

- 先验证本地播放测试
- 薄服务端返回：
  - `speak_text` 后由设备再请求 TTS
  - 或直接返回可播放音频 URL
- 统一输出格式，例如：
  - `16kHz mono PCM`
  - 或 `mp3`
- 播放前后更新墨水屏状态

交付物：

- 设备可播报服务端回复

## Phase 9: 功耗与可靠性

目标：变成真正可携带、可离线待机的设备。

任务：

- 待机时关闭不必要外设
- 合理使用 light sleep / deep sleep
- 降低墨水屏刷新频率
- 断网自动恢复
- 请求失败缓存最近状态
- 保存最后一屏内容，重启后立即恢复显示

交付物：

- 电池供电下可稳定待机与唤醒

## 服务端选型建议

短期建议：

- 薄服务端继续用 Python，开发最快
- 先不要做数据库和复杂账号系统
- 先把 `yesnoaudio/host/app_core.py` 中与转写/翻译相关的思路抽成 provider adapter 和 response shaper

推荐拆分：

- `server/app.py`
- `server/provider_adapter.py`
- `server/response_shaper.py`
- `server/tts_proxy.py`
- `server/schemas.py`

原因：

- 你现在已有参考逻辑
- 改造成本远低于重写一套
- API key 不需要固化在设备里
- 能最快把“设备直连服务器”跑通

## 关键风险

### 1. 板子版本风险

你的是 `V2`，示例和驱动必须严格匹配 `V2`，否则最容易在屏幕、音频、触摸初始化阶段卡住。

### 2. 音频链路复杂度

音频比屏幕和 Wi-Fi 难，尤其是：

- ES8311 初始化
- I2S 参数
- 麦克风输入路径
- 编码格式

所以必须先跑官方音频示例，再做网络上传。

### 3. 墨水屏刷新策略

墨水屏不适合高频 UI 动画。产品交互必须围绕“低刷新、清晰状态页”设计。

### 4. TLS 与证书

设备直连薄服务端后，HTTPS 证书、时间同步、连接失败重试会成为稳定性重点。

### 5. 内存与缓存

虽然 `V2` 有 `8MB PSRAM`，但录音缓存、屏幕缓存、JSON 解析、网络缓冲同时存在时仍要控制内存峰值。

## 建议的开发顺序

严格按下面顺序推进：

1. 跑通官方 `V2` 示例，确认屏幕 / Wi-Fi / 音频 / 传感器
2. 建立最小 ESP-IDF 工程与状态页
3. 做 SoftAP 配网和 NVS 保存
4. 跑通本地录音
5. 先做最小 Python 薄服务端
6. 设备通过 HTTPS 上传音频并拿回文本
7. 墨水屏显示结果
8. 再加温湿度、RTC、功耗优化
9. 最后再做 TTS 播放和更复杂交互

## 最终目标定义

当下面这条链路成立时，项目第一阶段算完成：

- 设备上电
- 自动连 Wi-Fi
- 用户按下按钮或触摸触发录音
- 设备把音频发到薄服务端
- 薄服务端调用第三方能力后返回文本结果
- 设备把结果显示到 200x200 墨水屏
- 全流程不依赖电脑常驻
